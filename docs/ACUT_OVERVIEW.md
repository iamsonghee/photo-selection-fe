# A컷 (ACUT) 서비스 개요

> 이 문서는 서비스 전체를 한눈에 파악하고 확장성·개선 방향을 논의하기 위한 참고 자료입니다.

---

## 1. 서비스 정의

**A컷**은 사진 작가와 고객 사이의 사진 선택(셀렉)·보정 납품 워크플로우를 디지털화한 SaaS 플랫폼입니다.

### 해결하는 문제

기존 방식: 작가가 수백 장의 원본을 Google Drive나 WeTransfer로 전달 → 고객이 엑셀·메모장으로 파일명 적어서 카카오톡으로 회신 → 작가가 수동으로 파일 찾아서 보정

A컷 방식: 작가가 업로드 → 고객이 링크로 갤러리 접속해서 클릭으로 선택 → 작가가 결과 바로 확인 후 보정·납품

### 주요 사용자

- **작가**: 웨딩·증명·프로필·가족 촬영 등 상업 사진 작가
- **고객**: 별도 가입 없이 링크만으로 사용 (access_token 기반)

---

## 2. 핵심 기능

### 작가 기능

| 기능 | 설명 |
|------|------|
| 프로젝트 생성 | 촬영 건별로 생성. 셀렉 갯수(N), 셀렉 기한, PIN, 재보정 횟수 설정 |
| 원본 업로드 | JPEG/PNG/WebP/HEIC 지원. 베타 최대 1,500장/프로젝트 |
| 고객 초대 링크 | N장 이상 업로드 완료 시 활성화. PIN 포함 복사 가능 |
| 셀렉 결과 확인 | 확정된 사진 목록 + 고객 코멘트 조회. CSV/TXT/클립보드 내보내기 |
| 보정본 업로드 | V1 · V2 최대 2라운드. 2MB 초과 시 자동 품질 조정 |
| 대시보드 | 진행 상태·D-day 한눈에 확인 |

### 고객 기능

| 기능 | 설명 |
|------|------|
| 링크 접속 | 회원가입·로그인 없이 접근 |
| 갤러리 | 가상 스크롤로 대량 사진 렌더링. 선택/해제 클릭 |
| 메타데이터 | 별점(1~5), 색상 태그, 코멘트 (선택 사항) |
| "선택됨" 탭 | 선택한 사진만 필터 |
| 최종 확정 | "보정 의뢰하기" → 모달에서 "확정 및 전송" |
| 확정 취소 | 작가 보정 시작 전 최대 3회 취소 가능 |
| 보정본 검토 | V1/V2 각 사진 확정 or 재보정 요청 + 코멘트 |

## 3. 워크플로우 (프로젝트 상태 흐름)

```
[작가] 프로젝트 생성
    ↓
preparing  — 원본 업로드 중 (업로드 수 < N)
    ↓  [작가: 초대 링크 활성화]
selecting  — 고객이 N장 선택 중
    ↓  [고객: "확정 및 전송"]
confirmed  — 셀렉 완료, 작가 보정 대기
    ↓  [작가: "보정 시작하기"]
editing    — 작가 V1 보정 중, 고객은 locked
    ↓  [작가: V1 전체 업로드 후 "고객에게 전달"]
reviewing_v1 — 고객 V1 검토 중
    ↓  [고객: 재보정 요청 있음]           [고객: 전부 확정]
editing_v2                                delivered (완료)
    ↓  [작가: V2 전달]
reviewing_v2
    ↓  [고객: 전부 확정]
delivered (완료)
```

**확정 취소 흐름**: `confirmed → selecting` (최대 3회, `cancel_count` 필드로 추적)

**재보정 0회 프로젝트**: `reviewing_v1` 없이 바로 납품. 고객은 원본/보정본 비교 후 "수령 완료"

## 4. 기술 스택

| 영역 | 기술 |
|------|------|
| 프론트엔드 | Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS 4 |
| 백엔드 | FastAPI (Python), Pillow + pillow-heif |
| 인증 | Supabase Auth (Google OAuth, JWT ES256) |
| DB | Supabase PostgreSQL (RLS 적용) |
| 스토리지 | Cloudflare R2 (CDN) |
| 배포 | 프론트: Vercel / 백엔드: Railway |
| 가상 스크롤 | TanStack Virtual |

### 주요 데이터 흐름

```
원본 업로드:
  프론트 → XHR Batch (PC: 8장×동시5 / 모바일: 3장×동시2)
  → 백엔드: EXIF 회전 보정 → 썸네일(400px) + 미리보기(1200px) 생성
  → R2 병렬 업로드 → Supabase DB INSERT

고객 갤러리:
  /c/[token] (인증 없음, access_token으로 프로젝트 조회)
  → SelectionContext: 선택 상태 전역 관리
  → 선택/해제: fire-and-forget API 호출 (즉각적 UX)
  → 확정: selected_photo_ids 포함 전송 → 서버에서 DB 교체 후 status 변경
```

### DB 핵심 테이블

| 테이블 | 역할 |
|--------|------|
| `photographers` | 작가 계정 (auth_id 연결) |
| `projects` | 프로젝트. status, required_count, cancel_count 등 |
| `photos` | 원본 사진. thumb_url, preview_url, R2 key |
| `selections` | 고객 선택 목록. rating, color_tag, comment 포함 |
| `photo_versions` | 보정본 파일 |
| `version_reviews` | 고객 보정본 검토 결과 |
| `project_logs` | 상태 변경 이력 |

## 5. 현재 제약 (베타 기준)

| 항목 | 제약 |
|------|------|
| 프로젝트 수 | 계정당 최대 10개 |
| 업로드 | 프로젝트당 최대 1,500장 |
| 재보정 | 최대 2라운드 (V1, V2) |
| 인증 | Google OAuth만 지원 (카카오 미구현) |
| 알림 | 자동 알림 없음, 링크 수동 공유 |
| 파일 형식 | JPEG/PNG/WebP/HEIC (RAW 미지원) |

---

## 6. 알려진 기술적 부채

| 항목 | 내용 |
|------|------|
| 갤러리 페이지네이션 | 현재 전체 사진 한 번에 로드 (500장+ 시 성능 저하 가능) |
| 알림 시스템 | 없음. 모든 단계 전환을 작가가 수동으로 고객에게 안내 |
| 파일명 특수문자 | URL 예약 문자 포함 파일명이 R2 key로 그대로 사용될 수 있음 |
| 실시간 동기화 | 선택 상태 폴링/웹소켓 없음 (새로고침 필요) |
| 모바일 업로드 UX | 모바일에서 대량 업로드 시 동시성 제한으로 느림 |

---

## 7. 확장 가능한 방향 (논의 포인트)

**비즈니스 확장:**
- 알림 시스템 (카카오 알림톡, 이메일)
- 구독 플랜 (프로젝트 수·용량 제한 기준)
- 팀 계정 (스튜디오 단위 관리)
- 고객 재접속 히스토리·분석

**기술 확장:**
- 갤러리 무한 스크롤 (현재 전체 로드)
- 웹소켓 실시간 선택 동기화
- 원본 RAW 파일 지원
- 보정본 직접 다운로드
- 모바일 앱 (현재 웹만)
- AI 셀렉 추천 (별점·선택 패턴 기반)
